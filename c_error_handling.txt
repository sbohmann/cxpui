## A solid, “fresh project” error-handling setup for C (incl. `fopen`)

In C you don’t get exceptions, so you usually pick a consistent *policy* and enforce it everywhere:

- **Leaf functions** return a status (`0/-1`, `bool`, or an `enum`) and set an “out” error if needed.
- **System/libc calls** are checked immediately (`NULL`, `-1`, etc.), and you capture `errno` right away.
- **Callers** decide what to do: retry, show a message, or propagate upward.
- Use a **single cleanup path** (`goto cleanup`) to avoid leaks.

Below is a pragmatic pattern that works well for most projects.

---

## 1) Define a small error type + helpers

This keeps errors uniform whether they come from your code or from libc.

```c
// error.h
#pragma once

#include <errno.h>
#include <stddef.h>

typedef enum {
    ERR_OK = 0,
    ERR_INVALID_ARG,
    ERR_IO,
    ERR_NOMEM,
    ERR_NOT_FOUND,
    ERR_INTERNAL
} err_code_t;

typedef struct {
    err_code_t code;   // your coarse category
    int sys_errno;     // copy of errno when relevant, else 0
    const char *msg;   // short static message (or NULL)
} err_t;

static inline err_t err_ok(void) {
    return (err_t){ .code = ERR_OK, .sys_errno = 0, .msg = NULL };
}

static inline err_t err_make(err_code_t code, const char *msg) {
    return (err_t){ .code = code, .sys_errno = 0, .msg = msg };
}

static inline err_t err_from_errno(err_code_t code, int e, const char *msg) {
    return (err_t){ .code = code, .sys_errno = e, .msg = msg };
}
```

And a tiny formatter for printing:

```c
// error.c
#include "error.h"
#include <stdio.h>
#include <string.h>

static const char *code_str(err_code_t c) {
    switch (c) {
        case ERR_OK: return "OK";
        case ERR_INVALID_ARG: return "INVALID_ARG";
        case ERR_IO: return "IO";
        case ERR_NOMEM: return "NOMEM";
        case ERR_NOT_FOUND: return "NOT_FOUND";
        case ERR_INTERNAL: return "INTERNAL";
        default: return "UNKNOWN";
    }
}

void err_fprint(FILE *out, err_t err) {
    if (err.code == ERR_OK) {
        fprintf(out, "OK\n");
        return;
    }
    if (err.sys_errno != 0) {
        fprintf(out, "%s: %s (errno=%d: %s)\n",
                code_str(err.code),
                err.msg ? err.msg : "(no message)",
                err.sys_errno,
                strerror(err.sys_errno));
    } else {
        fprintf(out, "%s: %s\n",
                code_str(err.code),
                err.msg ? err.msg : "(no message)");
    }
}
```

## 2) Wrap external calls (like `fopen`) so you *always* handle them the same way

`fopen` returns `NULL` on failure and sets `errno`. The key is: **read and store `errno` immediately** before you do anything else.

```c
// fileutil.h
#pragma once
#include <stdio.h>
#include "error.h"

err_t file_open(FILE **out, const char *path, const char *mode);
```

```c
// fileutil.c
#include "fileutil.h"
#include <errno.h>

err_t file_open(FILE **out, const char *path, const char *mode) {
    if (!out || !path || !mode) {
        return err_make(ERR_INVALID_ARG, "file_open: invalid argument");
    }

    *out = NULL;
    errno = 0;
    FILE *f = fopen(path, mode);
    if (!f) {
        int e = errno; // capture immediately
        // You can map certain errno values to different err_code_t if you like.
        return err_from_errno(ERR_IO, e, "fopen failed");
    }

    *out = f;
    return err_ok();
}
```

Why wrappers are worth it:
- You avoid repeating `if (!f) { int e = errno; ... }` everywhere.
- You enforce consistent messages + categories.
- You can later swap `fopen`→`fopen_s`-style variants, add logging, metrics, retries, etc.

---

## 3) Use a single cleanup path (`goto cleanup`) for resource safety

This is the least-bad way in C to guarantee cleanup without duplicating code.

```c
#include "fileutil.h"
#include "error.h"
#include <stdlib.h>

err_t load_first_line(char **out_line, const char *path) {
    err_t err = err_ok();
    FILE *f = NULL;
    char *line = NULL;

    if (!out_line || !path) {
        return err_make(ERR_INVALID_ARG, "load_first_line: invalid argument");
    }
    *out_line = NULL;

    err = file_open(&f, path, "r");
    if (err.code != ERR_OK) goto cleanup;

    line = malloc(1024);
    if (!line) {
        err = err_make(ERR_NOMEM, "malloc failed");
        goto cleanup;
    }

    if (!fgets(line, 1024, f)) {
        // fgets returns NULL on error or EOF; distinguish with ferror/feof
        if (ferror(f)) {
            err = err_from_errno(ERR_IO, errno, "fgets failed");
        } else {
            err = err_make(ERR_NOT_FOUND, "file is empty");
        }
        goto cleanup;
    }

    *out_line = line;
    line = NULL;

cleanup:
    if (f) fclose(f);
    free(line);
    return err;
}
```

---

## 4) Decide your project-wide conventions early

Pick one set and stick to it:

1. **Return style**
   - `err_t func(...)` (as shown) is very explicit.
   - or `int func(..., err_t *err)` if you want the fast-path to be a simple `0`.

2. **`errno` usage**
   - Only trust `errno` if the function indicates failure.
   - Copy it immediately: `int e = errno;`
   - Avoid calling other libc functions before copying it.

3. **Logging vs. reporting**
   - Usually: *library code returns errors; application code prints/logs*.
   - This prevents double-logging and keeps libraries reusable.

4. **Debug checks**
   - Use `assert()` for programmer bugs/invariants.
   - Use returned errors for runtime failures (I/O, network, allocation, user input).

## Two-tier error handling: *recoverable* vs *fatal*

What you want is a clean split:

1. **Recoverable / expected failures** (user input, optional files, “not found”, network timeouts): return an error and let callers decide.
2. **Fatal / unexpected failures** (programmer bugs, invariant violations, “this must exist”, out-of-memory you don’t want to handle, SDL init failure in a demo app): *crash politely*—print a message + backtrace, then exit.

That second tier is best done with **thin “must” wrappers/macros** so the main code stays uncluttered.

---

## A practical “fatal” facility (message + backtrace + exit)

On macOS, you can usually get a backtrace with `<execinfo.h>`.

```c
// fatal.h
#pragma once

#include <stdarg.h>

_Noreturn void fatalf(const char *file, int line, const char *func,
                      const char *fmt, ...);

void fatal_print_backtrace(void);

// Convenience macros
#define FATAL(...) fatalf(__FILE__, __LINE__, __func__, __VA_ARGS__)

#define MUST(cond, ...)                           \
    do {                                          \
        if (!(cond)) {                            \
            FATAL(__VA_ARGS__);                   \
        }                                         \
    } while (0)

#define MUST_NONNULL(p, ...) MUST((p) != NULL, __VA_ARGS__)
```

```c
// fatal.c
#include "fatal.h"

#include <execinfo.h>
#include <stdio.h>
#include <stdlib.h>

void fatal_print_backtrace(void) {
    void *frames[64];
    int n = backtrace(frames, (int)(sizeof(frames) / sizeof(frames[0])));
    backtrace_symbols_fd(frames, n, fileno(stderr));
}

_Noreturn void fatalf(const char *file, int line, const char *func,
                      const char *fmt, ...) {
    fprintf(stderr, "\nFATAL: %s:%d (%s): ", file, line, func);

    va_list ap;
    va_start(ap, fmt);
    vfprintf(stderr, fmt, ap);
    va_end(ap);

    fputc('\n', stderr);

    fprintf(stderr, "\nBacktrace:\n");
    fatal_print_backtrace();

    // Use abort() if you want a core dump, exit() if you want “cleaner” termination.
    abort();
}
```

### Notes
- `abort()` is great during development (debugger stops, crash report, core dumps if enabled).
- If you prefer controlled shutdown, use `exit(EXIT_FAILURE)`—but backtraces are typically more useful with `abort()`.

## “Must wrappers” for external APIs (examples: `fopen`, SDL, malloc)

### 1) `fopen` but fatal on failure
This keeps call sites clean while still showing *why* it failed (`errno`).

```c
// must_stdio.h
#pragma once
#include <stdio.h>

FILE *must_fopen(const char *path, const char *mode);
```

```c
// must_stdio.c
#include "must_stdio.h"
#include "fatal.h"

#include <errno.h>
#include <string.h>

FILE *must_fopen(const char *path, const char *mode) {
    errno = 0;
    FILE *f = fopen(path, mode);
    if (!f) {
        int e = errno; // capture immediately
        FATAL("fopen('%s','%s') failed: errno=%d (%s)",
              path, mode, e, strerror(e));
    }
    return f;
}
```

Usage becomes burden-free:

```c
FILE *cfg = must_fopen("config.ini", "r");
```

### 2) `malloc` but fatal on OOM (if that’s your policy)
```c
// must_alloc.h
#pragma once
#include <stddef.h>

void *must_malloc(size_t n);
```

```c
// must_alloc.c
#include "must_alloc.h"
#include "fatal.h"
#include <stdlib.h>

void *must_malloc(size_t n) {
    void *p = malloc(n);
    if (!p && n != 0) {
        FATAL("malloc(%zu) failed", n);
    }
    return p;
}
```

### 3) SDL-style wrappers (return codes + SDL_GetError)
SDL often provides its own error string. Wrap it once.

```c
// must_sdl.h
#pragma once
#include <SDL3/SDL.h>

void must_sdl_ok(int rc, const char *what);
void *must_sdl_ptr(void *p, const char *what);
```

```c
// must_sdl.c
#include "must_sdl.h"
#include "fatal.h"

void must_sdl_ok(int rc, const char *what) {
    if (rc < 0) { // SDL convention: <0 indicates failure for many calls
        FATAL("%s failed: %s", what, SDL_GetError());
    }
}

void *must_sdl_ptr(void *p, const char *what) {
    if (!p) {
        FATAL("%s failed: %s", what, SDL_GetError());
    }
    return p;
}
```

Usage:

```c
must_sdl_ok(SDL_Init(SDL_INIT_VIDEO), "SDL_Init");
SDL_Window *w = must_sdl_ptr(SDL_CreateWindow("Title", 800, 600, 0), "SDL_CreateWindow");
```

No repeated `if (!ptr) ...` noise.

## How to decide which calls get “must” vs “error return”

A simple rule that scales:

- Use **recoverable** returning errors when the *outside world* can cause it and you can continue:
  - user-provided file path
  - optional assets
  - network calls
  - parsing user config (report & exit *without* backtrace is often nicer)

- Use **fatal** when it indicates a bug, violated invariant, or “the program can’t meaningfully proceed”:
  - internal resource that should ship with the app but is missing
  - failing SDL initialization in a demo/game that can’t run without it
  - null pointer where your API contract says “never null”
  - `malloc` failure if you’ve chosen “no OOM recovery” (common in games/tools)

If you want to be extra clean: name them explicitly, e.g. `try_open_*` (recoverable) vs `must_open_*` (fatal).

(If you later want symbolicated stacks in release builds, that’s a bigger topic, but the above is the baseline.)

You’re right: **“expected vs unexpected” is a property of a *call site*, not a function**. If the callee bakes in “this is fatal” vs “this is recoverable”, you end up with the C equivalent of Java’s checked-exception pain—callers are forced into a policy they may not want.

The way out is to separate:

1. **Mechanism**: functions report failure in a uniform, lightweight way.
2. **Policy**: the caller chooses to *handle*, *propagate*, or *crash*.

## Pattern: always return an error; optionally “escalate” at the call site

### 1) Functions return `err_t` (+ out-params), never exit
```c
// file_open.c
#include <errno.h>
#include <stdio.h>
#include "error.h"

err_t file_open(FILE **out, const char *path, const char *mode) {
    if (!out || !path || !mode) return err_make(ERR_INVALID_ARG, "file_open: invalid arg");

    *out = NULL;
    errno = 0;
    FILE *f = fopen(path, mode);
    if (!f) {
        int e = errno;
        return err_from_errno(ERR_IO, e, "fopen failed");
    }
    *out = f;
    return err_ok();
}
```

### 2) Call site chooses: handle vs crash, without changing the callee

You add *adapters* (macros/helpers) that convert `err_t` into “fatal” when desired:

```c
// must.h
#pragma once
#include "fatal.h"
#include "error.h"

// Escalate an err_t to fatal, but only when the call site wants that policy.
#define MUST_OK(expr)                                                   \
    do {                                                                \
        err_t _e = (expr);                                              \
        if (_e.code != ERR_OK) {                                        \
            FATAL("%s (code=%d, errno=%d)",                             \
                  _e.msg ? _e.msg : "operation failed",                 \
                  (int)_e.code, _e.sys_errno);                          \
        }                                                               \
    } while (0)
```

Now the **same** `file_open()` supports both styles:

```c
#include "must.h"
#include "fileutil.h"

void parse_user_file(const char *user_path) {
    FILE *f = NULL;

    // expected failure: handle
    err_t e = file_open(&f, user_path, "r");
    if (e.code != ERR_OK) {
        // show a friendly message, keep running
        // (or return e to the caller)
        return;
    }

    // ...
    fclose(f);
}

void load_required_config(void) {
    FILE *f = NULL;

    // same function, but caller policy is "must succeed"
    MUST_OK(file_open(&f, "config.ini", "r"));

    // ...
    fclose(f);
}
```

No checked-exception-style “pollution”: the caller can either:
- explicitly check (when it’s worth handling), or
- use `MUST_OK(...)` (when it isn’t).

## Extending the same idea to “pointer-returning” APIs (SDL, malloc, etc.)

Some APIs return pointers rather than a status code. You can still keep policy at the call site by providing **two layers**:

1) a *try* wrapper that returns `err_t` + out pointer  
2) a *must* call-site macro that escalates

Example:

```c
// sdl_try.h
#pragma once
#include <SDL3/SDL.h>
#include "error.h"

err_t sdl_create_window(SDL_Window **out, const char *title, int w, int h, SDL_WindowFlags flags);
```

```c
// sdl_try.c
#include "sdl_try.h"

err_t sdl_create_window(SDL_Window **out, const char *title, int w, int h, SDL_WindowFlags flags) {
    if (!out) return err_make(ERR_INVALID_ARG, "sdl_create_window: invalid arg");
    *out = SDL_CreateWindow(title, w, h, flags);
    if (!*out) return err_make(ERR_INTERNAL, SDL_GetError()); // could store SDL error separately if you want
    return err_ok();
}
```

Call site:

```c
SDL_Window *win = NULL;
MUST_OK(sdl_create_window(&win, "Title", 800, 600, 0));  // fatal policy here
```

## Why this avoids the checked-exception trap

- **Callees do one thing**: report failures.
- **Callers decide**: ignore/handle/propagate/crash.
- You don’t encode “expectedness” into function signatures or naming.
- Yet you still keep code terse via `MUST_OK(...)` when you don’t want branching clutter.

```c
_Noreturn void fatalf(const char *file, int line, const char *func,
                      const char *fmt, ...) {
    fprintf(stderr, "FATAL %s:%d (%s): ", file, line, func);

    va_list ap;
    va_start(ap, fmt);
    vfprintf(stderr, fmt, ap);
    va_end(ap);

    fputc('\n', stderr);
    fprintf(stderr, "Backtrace:\n");
    print_backtrace();

    abort(); // or exit(EXIT_FAILURE) if you prefer
}
```

## 2) Example: `fopen` with both variants

### Error-returning version: `try_fopen` (result via out-param)
```c
// io.h
#pragma once
#include <stdio.h>
#include "error.h"

err_t try_fopen(FILE **out, const char *path, const char *mode);
FILE *xfopen(const char *path, const char *mode); // exiting variant
```

```c
// io.c
#include "io.h"
#include "fatal.h"

#include <errno.h>
#include <string.h>

err_t try_fopen(FILE **out, const char *path, const char *mode) {
    if (!out || !path || !mode) return err_make(ERR_INVALID_ARG, "try_fopen: invalid argument");

    *out = NULL;
    errno = 0;
    FILE *f = fopen(path, mode);
    if (!f) {
        int e = errno; // capture immediately
        return err_from_errno(ERR_IO, e, "fopen failed");
    }
    *out = f;
    return err_ok();
}
```

### Exiting version: `xfopen` (returns `FILE*` directly)
```c
// io.c (continued)
FILE *xfopen(const char *path, const char *mode) {
    FILE *f = NULL;
    err_t e = try_fopen(&f, path, mode);
    if (e.code != ERR_OK) {
        if (e.sys_errno != 0) {
            FATAL("xfopen('%s','%s'): %s (errno=%d: %s)",
                  path, mode,
                  e.msg ? e.msg : "failed",
                  e.sys_errno, strerror(e.sys_errno));
        }
        FATAL("xfopen('%s','%s'): %s", path, mode, e.msg ? e.msg : "failed");
    }
    return f;
}
```

Now the call site decides, with *no extra branching burden* when it doesn’t want it:

```c
// expected: user path may not exist
FILE *f = NULL;
err_t e = try_fopen(&f, user_path, "r");
if (e.code != ERR_OK) {
    // handle/report nicely
}

// required: must exist
FILE *cfg = xfopen("config.ini", "r");
```

## 3) Scaling the pattern: pointers and value-returning operations

### Pointer-returning external API (SDL, etc.)
You can do the same “two symbols” approach:

- `try_sdl_create_window(SDL_Window **out, ...) -> err_t`
- `xSDL_CreateWindow(...) -> SDL_Window*`

Key point: **the exiting variant can return the pointer directly**, which is exactly the ergonomics you want.

### Value-returning operations
If you need “return a value directly” but the try-version uses out-params, keep it consistent:

- `err_t try_read_u32(uint32_t *out, FILE *f);`
- `uint32_t xread_u32(FILE *f);`

The `x*` wrapper calls the `try_*` implementation and `FATAL(...)` on error.

---

## 4) Naming convention that stays pleasant

A common, readable scheme:

- `try_*` — may fail, return `err_t`
- `x*` — “eXits on failure” (from Plan 9 tradition), returns the natural result type

Examples:
- `try_fopen` / `xfopen`
- `try_malloc` / `xmalloc`
- `try_sdl_create_renderer` / `xSDL_CreateRenderer`

This keeps call sites very explicit: the *choice* is visible in the symbol name, not hidden in a macro or in function internals.
